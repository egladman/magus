# Durable memory across sessions and models

The session ends; the next one may be a different model, a different agent
host, or weeks later. `magus_memory` keeps three markdown files per repository
in the user's state directory - outside the repo (never committed, never
another contributor's problem) and shared across every git worktree and branch
of the repo. What a stronger model decided, a smaller model can read and apply.

Three files, three jobs:

| file        | job                                                            | write discipline                                     |
| ----------- | -------------------------------------------------------------- | ---------------------------------------------------- |
| `status`    | where the work stands NOW: current goal, next action, blockers | `op=write` (overwrite - it is a snapshot, not a log) |
| `progress`  | dated journal of what happened and what should happen next     | `op=append` (entries are date-stamped automatically) |
| `decisions` | decisions made, the WHY, and when to revisit                   | `op=append` (date-stamped automatically)             |

## Session start

1. `magus_memory` {file: "status"} - where did the last session leave off?
2. `magus_memory` {file: "decisions"} - what is already settled? Do not
   re-litigate a decision recorded here; if new evidence contradicts one, say
   so explicitly and record the reversal as a new dated entry.

Empty results just mean a fresh project - start writing.

## During and after work

- Finished or abandoned a meaningful unit of work: append one short progress
  entry - what changed, what is next, any tripwires found. Include the branch
  name when the work is branch-specific; the files themselves are shared
  across branches and worktrees by design, so entries carry that context.
- Pass a short `title` on every decisions append (and progress appends worth
  finding again): it folds into the dated heading, so scanning the headings
  previews the file before reading it.
- Made a choice another session would otherwise re-derive (architecture,
  naming, a rejected approach and why): append it to decisions. A bare
  "we chose X" helps nobody; the why is what the next session needs.
- Before ending a session: rewrite status so the next session's first read is
  current. A stale status is worse than none.

WRONG: dumping a transcript-sized narrative into progress.
CORRECT: three sentences - what changed, what is next, what to watch out for.

## Scope boundaries

- Intra-session working notes (checklists, partial findings) belong in
  `magus_scratchpad`, which is per-workspace and disposable - not here.
- Facts the repo already records (code structure, git history, MAGUS.md) do
  not belong in memory; query them fresh via magus_query instead.
- The files are plain markdown on disk; the tool result includes the path
  when a human wants to read or edit them directly.
